# Anomaly Report

This folder contains a structured anomaly report generated by an autoencoder-based
fraud detection model. It is intended for review by an agent or analyst looking for
indicators of fraud, waste, or abuse in Medicare/Medicaid billing.

Each subfolder is named for a single NPI and contains the following files.

---

## Files

### `provider_info.json`
Full NPPES registry record for this provider, including name, address, entity type,
taxonomy codes, and authorised official information. Also includes:
- `anomaly_score` — the model's reconstruction error score (higher = more anomalous)
- `anomaly_percentile` — what percentage of scored providers this NPI scores above
- `leie_label` — 1 if the provider is on the LEIE exclusion list, 0 otherwise

Use this to verify the provider's identity, check for address irregularities, confirm
the taxonomy matches the services billed, and cross-reference with other exclusion databases.

### `feature_comparison.csv`
One row per provider-level feature (derived statistics computed from monthly billing data).
Columns:
- `feature_value` — this provider's actual value for the feature
- `peer_median_value` — median value across all providers sharing the same primary taxonomy code
- `recon_error` — the autoencoder's reconstruction error for this feature (higher = more unexpected)
- `peer_median_recon` — median reconstruction error for the same-taxonomy peer group

Sorted by `recon_error` descending — the top rows are the features the model found
most anomalous. Large gaps between `feature_value` and `peer_median_value` on high-error
features are the strongest signals to investigate.

### `graphs/`
One PNG per provider-month feature, sorted by group reconstruction error (worst first,
reflected in the 3-digit filename prefix). Each graph shows the raw monthly time series
of that billing metric over the observation period.

Look for:
- Sudden spikes or step-changes in billing volume or payment amounts
- Consistently elevated values relative to what same-taxonomy peers typically show
- Trends that began or ended around a specific date (especially near the exclusion date if LEIE)
- Features with high `recon_error` (shown in the graph subtitle) that also show visual anomalies

The graph subtitle shows the group reconstruction error — a measure of how unexpected
this feature's behaviour is according to the model. Prioritise reviewing graphs with the
highest error values first.

---

## Suggested Review Order

1. Read `provider_info.json` to establish identity and flag anything unusual
   (e.g. address mismatches, taxonomy inconsistencies, multiple authorised officials).
2. Open `feature_comparison.csv` and focus on the top rows (highest `recon_error`).
   Large deviations from `peer_median_value` alongside high `recon_error` are the
   strongest fraud/waste signals.
3. Review the corresponding graphs in `graphs/` for those top features, looking for
   the visual patterns described above.
